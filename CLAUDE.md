# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`kubectl-gs` is the official Giant Swarm kubectl plug-in — a Go CLI installed as a `kubectl` subcommand (`kubectl gs ...`) and distributed via [Krew](https://krew.sigs.k8s.io/). It is the user-facing entry point to a Giant Swarm management cluster: OIDC login, templating CAPI cluster/app manifests, retrieving Giant Swarm resources, and managing a GitOps repo.

The Go module is `github.com/giantswarm/kubectl-gs/v6`, so internal imports carry the `v6` suffix.

## Common commands

The `Makefile` and `Makefile.gen.go.mk` are generated by [devctl](https://github.com/giantswarm/devctl) — edit the generator, not the files. Common targets:

- `make build` — build a local `kubectl-gs` binary (also: `make build-darwin-arm64`, `build-linux`, etc.).
- `make install` — `go build` and place the binary in `$(go env GOPATH)/bin`.
- `make run` — `go run -race main.go` (pass args via the usual `go run ... -- <args>` separator, or run the built binary).
- `make test` — `go test -race ./...`.
- `make lint` — `golangci-lint run -E gosec -E goconst --timeout=15m ./...`.
- `make imports` — `goimports -local github.com/giantswarm/kubectl-gs -w .`. CI rejects unsorted imports; run this before committing.
- `make fmt` / `make vet` — `go fmt` / `go vet`.

Tests rely on **golden files** (see `test/goldenfile/`). To regenerate them for a package, pass `-update`:

```
go test ./cmd/get/clusters -run Test_run -update
```

Run a single test the usual way: `go test ./cmd/get/clusters -run Test_run/case_2 -v`.

## Architecture

### Cobra command tree

`main.go` builds the root `*cobra.Command` via `cmd.New()` (`cmd/root.go`). The root wires up the top-level subcommands:

- `login` — OIDC + client-cert auth flows for management and workload clusters (`cmd/login/`, the largest subcommand; includes structured-auth detection, workload-cluster routing, kubeconfig munging).
- `template` — render manifests for `cluster`, `app`, `catalog`, `organization`. Cluster templating fans out per provider under `cmd/template/cluster/provider/{capa,capv,capvcd,capz,eks}.go`, with Go `text/template` assets in `cmd/template/cluster/provider/templates/`.
- `get` — list/show Giant Swarm resources (`apps`, `catalogs`, `clusters`, `nodepools`, `orgs`, `releases`, `capi`).
- `gitops` — scaffold and modify a GitOps repo (`add`, `initialize` subtrees under `cmd/gitops/`).
- `validate`, `deploy`, `update`, `selfupdate`, `credentialplugin` — smaller utility commands.

### The subcommand pattern

Every subcommand follows the same four-file shape and should keep doing so:

```
cmd/<group>/<name>/
  command.go   // exports Config{} + New(Config) (*cobra.Command, error); wires flags + runner
  flag.go      // *flag struct with Init(cmd) + Validate(); pflag bindings
  runner.go    // type runner; Run(cmd, args) wraps run(ctx, cmd, args) with microerror.Mask
  error.go     // microerror.New + IsXxx matchers
```

`command.go` validates its `Config` (returning `invalidConfigError`), constructs the runner, and registers cobra metadata (Use/Short/Long/Example, Aliases, Args, RunE, PreRunE for middleware). Persistent kubeconfig flags come from `genericclioptions.RESTClientGetter` (initialized in `cmd/flag.go` and threaded down via each `Config.ConfigFlags`).

### Data-domain services

Behind the `get` and template commands sits a small service layer at `pkg/data/domain/<resource>/`:

```
pkg/data/domain/cluster/
  spec.go      // Interface, GetOptions, Resource, Collection
  service.go   // implementation backed by controller-runtime client
  error.go
```

A runner instantiates the service via `commonconfig.CommonConfig` (which holds the `RESTClientGetter` and lazily resolves the installation/provider via `pkg/installation` and Athena). Use the `Interface` type in tests and inject a fake client built from `pkg/scheme/scheme.go` — that scheme registers Application (App, Catalog), Giant Swarm core (Spark), Release SDK, organization-operator, and apiextensions.

### Provider abstraction

Provider names are constants in `internal/key/provider.go`: `aws`, `azure`, `capa`, `capz`, `eks`, `vsphere`, `cloud-director`. Two helper functions encode capability sets the rest of the code switches on:

- `PureCAPIProviders()` — fully migrated to CAPI (capa, capz, eks, vsphere, cloud-director).
- `CAPIProvidersUsingReleases()` — CAPI providers backed by Giant Swarm `Release` CRDs (capa, capz, vsphere, cloud-director).

When adding a provider, update both lists if applicable, the scheme if a new CRD is involved, and the corresponding template/get switch statements.

### Middleware

Cobra `PreRunE` hooks come from `pkg/middleware`. `middleware.Compose(...)` chains middlewares; the main one in use is `renewtoken.Middleware` which refreshes the OIDC token before the request runs.

### Output and golden tests

`pkg/output` defines output formats (`default`, `name`, `json`, `yaml`) plumbed through `genericclioptions.PrintFlags`. Tests typically capture stdout into a buffer and diff against a `.golden` file under each command's `testdata/` directory. `test/goldenfile/` is the helper, `test/kubeclient/` and `test/kubeconfig/` provide fake clients and kubeconfigs.

### Cross-cutting

- **Errors**: always `microerror.Mask(err)` on the way out and `microerror.Maskf(typedErr, "...")` for new ones declared in the package's `error.go`. Match with `IsXxx` helpers.
- **Logging**: `micrologger` from `cmd/root.go` is wrapped with an activation logger so logs are emitted only at error/verbose-1 by default; the `--debug` arg flips stack traces on via `pkg/errorprinter`.
- **Selfupdate**: `cmd/runner.go`'s `PersistentPostRunE` does a version check on every command run (except `selfupdate` itself), gated by `--disable-version-check`.
- **Telemetry**: `PersistentPreRun` sends a `GiantSwarm.command` signal via `telemetrydeck-go`. Disable with `KUBECTL_GS_TELEMETRY_OPTOUT=1`.
- **Version embedding**: `pkg/project.{version,buildTimestamp,gitSHA}` are injected via `-ldflags` from the Makefile (see `LDFLAGS`).

## Repo-specific conventions

- **CHANGELOG.md is mandatory.** New entries go under `## [Unreleased]` using Keep-a-Changelog sections (`Added`, `Changed`, `Fixed`, `Removed`). Match the existing terse style — lead with the affected subcommand, e.g. ``- `login`: ...``.
- **Conventional commits + semver.** Release flow is in `docs/Release.md`: branch shape `master#release#v<MAJOR.MINOR.PATCH>` triggers the release PR workflow.
- **`zz_generated.*` files are generator output** (devctl-managed) — `.cursor/rules/zz_generated.*.mdc`, `.github/zz_generated.windows-code-signing.sh`, the Makefiles. Don't hand-edit; change the template upstream.
- **Style guide.** The cursor rules point at <https://github.com/giantswarm/fmt/> as the canonical Go style/changelog/release reference for Giant Swarm code.
